# Use GitHub Issues as orchestration state

The orchestrator stores lifecycle state in GitHub Issues, labels, branches, and worktrees instead of introducing a separate database for v1. PRD Issues carry preparation metadata such as the PRD branch, and Implementation Issues carry claim metadata such as claim time, agent run identity, and implementation branch. This keeps the system recoverable from the repository and issue tracker while avoiding a second state store until analytics or stronger leasing justify it. If a GitHub issue-state Lifecycle Mutation fails part-way through, the operation is treated as an ambiguous lifecycle state: the orchestrator stops the operation, emits diagnostics, and relies on reconciliation or human review to recover from the observed issue state rather than continuing as if the intended state transition completed. In-memory Agent Run Capacity tracking is transient and represents only Agent Runs currently executing inside one orchestrator process; durable claim state in GitHub Issues remains the source of truth for stale-claim and retry handling. Recoverable operational failures may be skipped for forward progress, but they must still be surfaced as scheduler events, dashboard diagnostics, or issue lifecycle diagnostics according to their effect. The Orchestration Control Center should return degraded per-item diagnostics for malformed issues or unreadable run state, reserving whole-response failures for collection-level failures such as being unable to list issues.
